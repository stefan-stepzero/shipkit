Overview: Plugin vs Skills Ecosystem

Claude Code has two distribution channels for extending functionality:

Aspect Plugins (/plugin) Skills (npx skills)
Built into Claude Code itself External CLI (skills.sh)
Contains Commands, agents, skills, hooks, MCP servers Skills only (SKILL.md files)
Source Marketplaces (catalogs of plugins) Direct from repos
Installation Two-step: add marketplace → install plugin One-step: add skill
Namespacing Namespaced (e.g., plugin:skill-name) Flat namespace

Git & PR Workflow Plugins

commit-commands

Productivity No Shipkit Overlap
/commit /commit-push-pr /clean_gone
What it does:
  • /commit - Analyzes staged/unstaged changes, examines recent commit messages for style, drafts appropriate message, stages files, creates commit
  • /commit-push-pr - Complete workflow: creates branch (if on main), commits, pushes, creates PR with summary & test plan
  • /clean_gone - Removes local branches deleted from remote, handles worktrees
Key features:
  • Matches repository's commit message style automatically
  • Avoids committing secrets (.env, credentials)
  • PR descriptions include summary bullets and test plan
  • Requires: gh CLI for PR creation
Recommendation: DEFAULT INTEGRATE - Essential git workflow, no Shipkit conflict, everyone needs this.

code-review

Productivity No Shipkit Overlap
/code-review [--comment] + 4 parallel agents
How it works:
  1. Checks if review needed (skips closed, draft, trivial, already-reviewed)
  2. Gathers CLAUDE.md guidelines from repository
  3. Launches 4 parallel agents:
    • Agents #1 & #2: Audit for CLAUDE.md compliance
    • Agent #3: Scan for obvious bugs in changes
    • Agent #4: Analyze git blame/history for context
  4. Scores each issue 0-100 for confidence
  5. Filters out issues below 80 threshold
  6. Outputs to terminal or posts as PR comment (--comment)
Confidence scoring:
0Not confident, false positive
25Somewhat confident, might be real
50Moderately confident, real but minor
75Highly confident, real and important
100Absolutely certain
Recommendation: RECOMMEND IN GET-SKILLS - Great for teams, but heavy (4 parallel agents = token cost). Let user opt-in.

pr-review-toolkit

Productivity No Shipkit Overlap
/pr-review-toolkit:review-pr + 6 specialized agents
The 6 agents:
Agent Focus Triggers
comment-analyzer Comment accuracy, documentation, comment rot "Check if comments are accurate"
pr-test-analyzer Test coverage quality, critical gaps "Check if tests are thorough"
silent-failure-hunter Silent failures in catch blocks, error handling "Review error handling"
type-design-analyzer Type encapsulation, invariants (rated 1-10) "Review the UserAccount type design"
code-reviewer CLAUDE.md compliance, style, bugs "Review my recent changes"
code-simplifier Complexity, redundancy, clarity "Simplify this code"
Recommendation: RECOMMEND IN GET-SKILLS - Deep analysis for power users. 6 agents is expensive. Optional.

Feature Development Plugins

feature-dev

Development HIGH SHIPKIT OVERLAP
/feature-dev code-explorer code-architect code-reviewer
The 7-Phase Workflow:
Phase Goal What Happens
1. Discovery Understand what to build Clarifies request, asks about problem, constraints
2. Codebase Exploration Understand existing code Launches 2-3 code-explorer agents in parallel, identifies key files
3. Clarifying Questions Fill gaps, resolve ambiguity Asks about edge cases, error handling, integration, waits for answers
4. Architecture Design Design approaches Launches 2-3 code-architect agents with focuses: minimal, clean, pragmatic. Asks which you prefer
5. Implementation Build the feature Waits for explicit approval, follows chosen architecture
6. Quality Review Ensure quality Launches 3 code-reviewer agents (simplicity, bugs, conventions), presents findings
7. Summary Document results What was built, decisions made, files modified, next steps
Example interaction:
/feature-dev Add user authentication with OAuth

# Claude asks clarifying questions:
# - Which OAuth providers? (Google, GitHub, custom?)
# - Store OAuth tokens or just user profile?
# - Replace current auth or add alongside?

# Then presents 3 architecture approaches with trade-offs
# Asks which you prefer before implementing
Recommendation: EVALUATE CONFLICT - Significant overlap with Shipkit's spec→plan→implement flow. See detailed analysis below.

frontend-design

Development Fills Gap
Skill (auto-invoked for frontend work)
What it does:

Automatically injects guidance when Claude detects frontend/UI work. Prevents "AI slop" aesthetics.

Design philosophy injected:
  • Typography - Distinctive fonts, avoid generic (Inter, Roboto, Arial)
  • Color - Dominant colors with sharp accents, not timid palettes
  • Motion - High-impact animations, staggered reveals, scroll triggers
  • Spatial - Unexpected layouts, asymmetry, grid-breaking
  • Backgrounds - Gradient meshes, noise textures, decorative elements
What it prevents:
  • Overused fonts (Inter, Roboto, Arial, system fonts)
  • Cliched color schemes (purple gradients on white)
  • Predictable layouts and cookie-cutter patterns
Recommendation: DEFAULT INTEGRATE - Fills a gap in Shipkit (no UI guidance). Auto-invoked, low overhead.

Automation & Hooks

hookify

Productivity Advanced Users
/hookify /hookify:list /hookify:configure /hookify:help
What it does:

Creates hooks from conversation patterns via simple markdown files. No coding required.

Example rule file (.claude/hookify.warn-rm.local.md):
---
name: block-dangerous-rm
enabled: true
event: bash
pattern: rm\s+-rf
action: block
---

⚠️ **Dangerous rm command detected!**

This command could delete important files.
Events supported:
  • bash - Triggers on Bash commands
  • file - Triggers on Edit, Write, MultiEdit
  • stop - Triggers when Claude wants to stop
  • prompt - Triggers on user prompt
  • all - All events
Recommendation: RECOMMEND IN GET-SKILLS - Powerful for advanced users who want custom hooks. Could complement Shipkit's hooks.

ralph-wiggum

Development Niche
/ralph-loop /cancel-ralph + Stop hook
What it does:

Self-referential AI loops - keeps running same prompt until completion promise detected.

/ralph-loop "Build a REST API for todos. Requirements: CRUD, validation, tests. Output <promise>COMPLETE</promise> when done." --completion-promise "COMPLETE" --max-iterations 50
How it works:
  1. You run /ralph-loop once with a prompt
  2. Claude works on the task
  3. Tries to exit → Stop hook blocks it
  4. Stop hook feeds SAME prompt back
  5. Repeat until completion promise found
Real-world results cited:
  • 6 repositories generated overnight (Y Combinator hackathon)
  • $50k contract completed for $297 in API costs
Recommendation: RECOMMEND IN GET-SKILLS - Niche but powerful for autonomous overnight runs.

security-guidance

Security Safety Net
PreToolUse hook
What it does:

Warns about potential security issues when editing files. Monitors 9 security patterns:

  • Command injection
  • XSS vulnerabilities
  • eval() usage
  • Dangerous HTML
  • Pickle deserialization
  • os.system calls
  • SQL injection patterns
  • Hardcoded credentials
  • Unsafe file operations
Recommendation: DEFAULT INTEGRATE - Valuable safety net, runs silently as hook, no user overhead.

Focuses on: implementation choices, patterns, trade-offs, codebase-specific details.

learning-output-style

Learning Pedagogical
SessionStart hook (includes explanatory features)
What it does:

Requests user write 5-10 lines of meaningful code at decision points. Transforms "watch and learn" into "build and understand".

When Claude requests contributions:
  • Business logic with multiple valid approaches
  • Error handling strategies
  • Algorithm choices
  • Design patterns
When Claude won't request:
  • Boilerplate code
  • Obvious implementations
  • Configuration/setup
Recommendation: NICHE/SKIP - Pedagogical mode, not for production work.

Feature-dev vs Shipkit Detailed Analysis

The feature-dev plugin has significant overlap with Shipkit's workflow:

feature-dev Phases

  1. Discovery (understand requirements)
  2. Codebase Exploration (2-3 parallel agents)
  3. Clarifying Questions (waits for answers)
  4. Architecture Design (3 approaches)
  5. Implementation (waits for approval)
  6. Quality Review (3 agents)
  7. Summary (documentation)

Shipkit Equivalents

  1. /shipkit-product-discovery, /shipkit-spec
  2. (Claude does naturally)
  3. /shipkit-spec (requirements)
  4. /shipkit-plan, /shipkit-architecture-memory
  5. (Claude does naturally)
  6. /shipkit-verify, /shipkit-preflight
  7. /shipkit-work-memory

Key Differences

Aspect feature-dev Shipkit
Focus Code features Product development
Multi-agent Yes (2-3 parallel agents per phase) No (single agent flow)
Persistence Session only Files in .shipkit/
Architecture options 3 approaches with trade-offs Single recommended approach
Approval gates Explicit (waits for user) Implicit (continues unless stopped)

Resolution Options

Option A: Keep Both

Use Shipkit for product work (why, who, what), feature-dev for pure code features. Document when to use each.

Option B: Replace Shipkit Workflow

Adopt feature-dev as the implementation workflow. Risk: loses Shipkit's product/user focus.

Option C: Enhance Shipkit

Borrow patterns from feature-dev: multi-agent exploration, architecture options, explicit approval gates. Integrate into existing Shipkit flow.
